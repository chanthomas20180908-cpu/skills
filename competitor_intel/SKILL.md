## 依赖与前置条件

### 强烈推荐安装（获得最佳调研效果）

- `web-access`：统一处理所有联网操作，especially 动态页面、登录态、社媒、广告库。
- `multi-search`：多引擎搜索，自动切换 DuckDuckGo / Tavily / Bing。
- `browse`：浏览器 QA，适合需要真实浏览器环境的复杂页面。

未安装上述 Skill 时，本 Skill 仍可运行，但动态信源（X、YouTube、LinkedIn、TikTok、Instagram、Meta Ads、Google Ads）通常只能标记为“未覆盖”。

### 最低环境要求

- Node.js 18+（`update_monitoring_state.mjs` 使用 `import.meta.dirname`）。
- `curl` 二进制（脚本在 `fetch` 失败时回退到 `curl`）。
- 支持 Bash 的 coding agent。

# 竞品增长情报双周报

## 核心原则

- **固定周期**：每次独立调查指定的双周时间窗，不与上一次报告比较。
- **事实记录**：只写来源能够直接证明的已发生内容，不推断目的、目标人群、战略意图或预期效果。
- **时间优先**：先形成事件清单，再按可证明的发生时间升序整理。
- **完整覆盖**：所有必查信源都必须执行并留下台账；不能只采集官网和社区后结束。
- **双源核验**：每个事件必须寻找主信源和二次信源，并明确两者是否独立。
- **确认边界**：来源只能证明“更新”时，不得写成“首次上线”；只能证明“页面存在”时，不得写成“本期发布”。
- **来源透明**：正文记录逐项证据，报告底部提供信源总表、采集执行台账和覆盖缺口。

## 时间口径

每个事件区分以下时间：

- `发生时间`：来源能够直接证明事件发生的日期或时间。
- `来源发布时间`：文章、公告、帖子或页面记录显示的发布时间。
- `来源更新时间`：sitemap、changelog 或页面元数据显示的更新时间。
- `发现时间`：本次调研访问该来源的时间。

规则：

1. 按 `发生时间` 升序排列。
2. 没有发生时间时，可用明确的来源发布时间；必须标注所采用的时间口径。
3. 只有更新时间时，事件名称必须使用“更新”“出现更新时间”等准确表述。
4. 日期无法确认的事件放入“时间未确认事件”，不混入主时间线。
5. 周期外事件不进入正文；只有解释本期事件必需时，才在事件卡的“背景事实”中简述并附来源。

## 信源与二次确认

### 主信源

直接承载事件的原始材料，例如：

- 官方公告、官方博客文章、产品页面、定价页、changelog、帮助中心
- 官方 sitemap、RSS、结构化数据、广告透明度记录
- 原始社交帖子、原始社区帖子、原始视频、原始 release
- Creator 原始内容、合作方原始公告、用户原始评论

### 二次信源与来源关系

每个二次信源必须标记以下关系之一：

- `独立信源`：不同组织或独立发布者形成的第二份原始证据，例如官方公告 + 媒体原创采访、品牌原帖 + Creator 原帖、广告库记录 + 实际落地页。
- `同源佐证`：同一组织控制的另一页面或记录，例如官方文章 + 官方 Blog 索引、产品页 + 官方 sitemap、官方 X + 官方 YouTube。
- `转述/副本`：同一新闻稿转载、聚合复述、缓存副本或无新增核验的引用。

硬规则：

1. 只有 `独立信源` 可以构成“✅ 已二次确认”。
2. `同源佐证` 可以确认页面存在、日期或官方口径，但最高只能标记“⚠️ 部分确认”。
3. `转述/副本` 不构成二次确认，事件仍为“◯ 单一来源”。
4. 第二个 URL 不等于第二个独立信源，必须判断组织归属和证据是否新增。
5. 用户评论、Creator 评价和媒体评价只能证明其内容被发布，不能直接证明产品客观性能。

### 确认状态

- `✅ 已二次确认`：至少两份组织归属独立的证据共同支持事件核心事实。
- `⚠️ 部分确认`：同源佐证成立，或独立二次信源只确认事件的一部分，或时间只能确认到更新时间。
- `◯ 单一来源`：只有一份有效原始证据；转述和副本不提升确认级别。
- `❌ 来源冲突`：来源对核心事实存在冲突。

报告不得因为缺少独立二次信源而删除真实事件，但必须降级并写清未确认内容。

## Fallback：未安装 web-access 时的采集策略

如果当前环境没有安装 `web-access` 或其他浏览器 Skill，按以下优先级回退：

1. **静态页面 / sitemap / RSS**：使用 `curl` 或 `WebFetch` 直接拉取。
2. **搜索发现**：使用 `WebSearch` 执行 `site:{domain}` 和品牌/别名/动作关键词组合搜索。
3. **Reddit / Hacker News / GitHub / Product Hunt / 评论平台**：优先使用公开 API 或静态页面；无法访问时标记“未覆盖”。
4. **动态 / 登录态 / 反爬强的平台**（X、YouTube、LinkedIn、TikTok、Instagram、Meta Ad Library、Google Ads Transparency Center）：
   - 如果 `web-access` / `browse` 不可用，必须在该渠道的采集执行台账中标记为“未覆盖”。
   - 不得使用未经验证的二手摘要、AI 生成内容或猜测填充事件。

所有 fallback 采集同样必须写入执行台账，并遵守“事实口径”和“确认边界”规则。

## Step 0：确定范围

1. 读取 `competitors.yaml`，确认目标竞品、GIQ、品牌别名、域名和信源配置。
2. 优先使用用户或任务明确给出的周期起止日期。
3. 若只给出结束日期，则按包含首尾的 14 个自然日计算，起始日期为结束日期减 13 天，例如 `2026-05-30 ~ 2026-06-12`。
4. 本次调查仅以时间窗为筛选条件，不读取 state 作为事件纳入依据。
5. 建立本轮 `source_execution_ledger`，先写入全部必查渠道，再开始采集。
6. 记录本次发现时间。
7. `update_monitoring_state.mjs` 当前仅内置 `neuralframes` 与 `freebeat` 的自动化监控；`vidmuse` 与 `tunee` 需依赖 Step 1 的手动采集。

## Step 1：按信源并行采集

按信源组启动独立子 Agent。每个子 Agent 必须：

- 执行该组列出的最低检索动作，不能只搜索品牌名一次。
- 只返回时间窗内的候选事件；周期外结果仅进入排除项。
- 返回原始 URL、页面标题、可证明日期、原文直接支持的事实。
- 区分“事件事实”和“研究者判断”；后者不得进入结果。
- 为每个候选事件提出独立二次核验路径。
- 将每次检索写入采集执行台账。
- 搜索成功但无合格结果时标记“已覆盖，未采集到事件”。
- 页面、权限、登录态或反爬导致无法核验时标记“未覆盖”，不得写成“无动作”。

### 1A：官网、产品与自有内容

必查信源：

- 首页、产品页、定价页、Blog、Changelog / Release Notes、Help Center / Docs
- Sitemap、RSS、页面结构化数据和公开 Newsletter 归档
- 新工具页、对比页、场景页、use-case 页、Marketplace / Integrations 页

最低检索动作：

1. 逐个访问配置中的官方入口。
2. 解析 sitemap / RSS 中时间窗内的发布日期或更新时间。
3. 使用 `site:{domain}` 搭配品牌、产品、pricing、release、update、launch 等词补查未进入导航的页面。
4. 定价只在取得历史证据时判断“变化”；只有当前快照时只记录当前状态，不生成变价事件。
5. 页面 `lastmod` 只证明记录的更新时间，不证明首次发布。

### 1B：官方社媒与视频

必查信源：

- X / Twitter
- YouTube
- LinkedIn 公司页及已配置的创始人/核心成员账号
- TikTok
- Instagram

最低检索动作：

1. 检查配置中的官方账号时间线。
2. 分别检索品牌名、品牌别名、官网域名和产品名，并限定时间窗。
3. 检查发布、功能演示、合作方或 Creator 标记、活动、优惠及高互动内容。
4. 视频和帖子记录发布时间、原始链接、账号身份、内容事实；互动数字必须记录抓取时间。
5. 官方多个社媒账号相互印证仍属于“同源佐证”。

### 1C：社区、评论与 Creator 内容

必查信源：

- Reddit 全站搜索及配置的 Subreddit
- Hacker News
- G2、Trustpilot、Capterra、Product Hunt 评论
- YouTube / TikTok / Instagram 的第三方 Creator 内容
- App Store / Google Play（产品存在移动应用时）

最低检索动作：

1. 使用品牌名、别名、域名和主要产品名进行全站搜索。
2. Reddit 不能只查配置中的 Subreddit；必须同时执行全站检索。
3. 评论平台筛选周期内新增内容，记录评论日期、评分、原文和总量快照。
4. 检查 Creator 内容中的品牌链接、优惠码、`sponsored`、`paid partnership` 或明确合作披露。
5. 单一用户或 Creator 观点只能形成”帖子/评论已发布”事件，不能升级为产品事实。
6. 符合 `community_signal` 入账硬规则的 Reddit 帖子、评论、教程**必须入主时间线**，不得直接放入 §3。

### 1D：发布、分发、合作与媒体

必查信源：

- Product Hunt 产品页、Launch、Maker 评论
- Marketplace、插件商店、集成目录
- 合作方官网及合作方社媒
- 媒体原创报道、采访、Podcast 和行业 Newsletter
- 融资或公司资料平台（仅在 GIQ 涉及经营数据时）

最低检索动作：

1. 搜索品牌名 + `launch`、`release`、`partnership`、`integration`、`funding`、`ARR`、`revenue`、`interview`。
2. 优先追溯到官方原文、合作方原文或媒体原创内容。
3. 同一新闻稿的多站转载合并为一条线索，不计多个信源。
4. 经营数据保留归属，例如“公司向某媒体披露”，不得写成独立审计事实。

### 1E：付费广告

必查信源：

- Meta Ad Library
- Google Ads Transparency Center

最低检索动作：

1. 使用品牌名、广告主页、广告主名称和官网域名分别检索。
2. 记录广告 ID、投放状态、首次展示时间、地区、文案、素材、CTA 和落地页。
3. 广告库记录作为主信源，实际落地页作为事件内容的佐证；二者组织独立性须单独判断。
4. 动态页面、登录态或地区限制导致失败时，台账标记“未覆盖”并写明原因。
5. 不得使用搜索引擎中的广告文案摘要替代广告库原始记录。

### 1F：开发者与公开技术记录

按适用性检查：

- GitHub / GitLab releases、公开仓库、issues、package registry
- API 文档、SDK changelog、浏览器扩展商店
- Status Page、公开事故记录

闭源或无公开入口时标记“不适用”，并记录判断依据；不能简单省略。

## Step 2：建立采集执行台账

每个必查渠道至少有一条执行记录：

```text
source_id
source_group
platform
account_or_entry
query_or_action
period
executed_at
result_count
status
failure_reason
candidate_event_ids[]
```

`status` 只能使用：

- `已覆盖-发现事件`
- `已覆盖-未采集到事件`
- `部分覆盖`
- `未覆盖`
- `不适用`

规则：

- `result_count` 是检索返回或人工核验的候选数量，不等于最终事件数量。
- `部分覆盖` 必须说明缺少哪个账号、时间段、结果页或登录态。
- `未覆盖` 必须写访问失败、权限、验证码、动态渲染、地区限制等具体原因。
- 未写入执行台账的渠道一律视为“未执行”，报告不得宣称完成完整调研。

## Step 3：建立事件账本

合并候选结果，并为事件分配 `E01`、`E02` 等 ID。

每个事件必须包含：

```text
event_id
event_name
category
occurred_at
time_basis
discovered_at
facts[]
primary_source
secondary_source
source_relationship
verification_status
verified_scope
unverified_scope
```

合并规则：

- 同一日期、同一发布动作下的多个页面可合并为一个事件，但必须列出完整清单。
- 同一内容被多个媒体转载，不拆成多个事件。
- 产品事实与用户评价是不同事件，不得混写。
- 没有足够内容支撑结构化事件卡的线索，放入”未纳入线索”，说明原因。

### 事件 category 定义

| category | 含义 | 典型例子 |
|---|---|---|
| `seo_content` | 新增或更新的 SEO 落地页、博客、帮助中心文章 | 新工具页、对比页、use-case 页 |
| `product_update` | 功能发布、版本更新、模型迭代 | changelog 条目、新功能公告 |
| `pricing_change` | 定价调整、Free tier 变更、促销折扣 | 价格页对比快照 |
| `paid_ad` | Meta / Google 广告素材或落地页变更 | 广告库新素材 |
| `social_content` | 官方社媒原创发布（非转发） | 官方 X/Instagram/TikTok 帖子 |
| `creator_partnership` | Creator 明确披露赞助、Affiliate 码、付费合作 | `#sponsored`、优惠码、合作方公告 |
| `community_signal` | 社区自发讨论、第三方评论、非赞助教程 | Reddit 帖子、Trustpilot 评论、第三方 YouTube 教程 |
| `launch_distribution` | Product Hunt 上线、Marketplace 入驻、合作方公告 | PH Launch、集成目录新增 |
| `media_coverage` | 媒体原创采访、Podcast、行业 Newsletter | TechCrunch 报道 |

**`community_signal` 入账硬规则：**

1. Reddit 帖子含品牌名且有实质讨论（≥3 条回复）→ 必须入账，不得放入 §3。
2. Trustpilot / G2 / Capterra 周期内新增评论 → 记录数量快照、评分分布、代表性原文（1-2 条）。
3. 第三方 Creator 发布教程、测评、对比视频 → 记录标题、URL、发布时间、订阅量/播放量快照。
4. `community_signal` 的 `facts[]` 只写”内容已发布”事实，不写产品性能断言。
5. 若 Creator 内容含赞助披露，升级为 `creator_partnership` 类别。

- [ ] 时间线按发生时间升序？
- [ ] 每个事件都有结构化“已发生内容”？
- [ ] 每个事件都列出主信源、二次信源和来源关系？
- [ ] 只有组织独立的证据才标记“✅ 已二次确认”？
- [ ] 官方文章 + 官方索引等同源组合是否降为“⚠️ 部分确认”？
- [ ] 明确区分发布时间、更新时间和发现时间？
- [ ] 没把页面更新时间写成首次上线时间？
- [ ] 没有目的、人群、战略意图或效果推断？
- [ ] 官网/产品、X、YouTube、LinkedIn、TikTok、Instagram、Reddit、HN、Product Hunt、Creator/评论、媒体/合作、Meta Ads、Google Ads 是否都有执行台账？
- [ ] 不适用渠道是否写明判断依据？
- [ ] 采集失败是否标为“未覆盖”而非“无动作”或“未采集到”？
- [ ] 底部信源总表覆盖正文全部链接？
- [ ] 覆盖摘要数字是否与执行台账一致？
- [ ] 报告文件路径和命名正确？

任何一个必查渠道缺少执行台账时，不得将报告标记为“完整覆盖”；必须在报告顶部写明“覆盖不完整”。

## State 使用边界

`state/` 可以保留用于采集缓存、断点续抓和减少重复网络请求，但：

- 不决定事件是否纳入报告。
- 不作为“本期新增”的证明。
- 不要求报告与上次报告比较。
- 报告纳入条件只有两个：事件时间位于本期，且存在可审计来源。
