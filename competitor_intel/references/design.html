01 · 战略定位

这个工具解决什么问题

核心目标：在固定双周时间窗内，系统记录竞品在获客侧和产品侧实际发生的动作。报告按时间排序，每个事件提供主信源、二次信源、确认状态与确认边界。

🔴 P0 · 获客情报

竞品怎么把人拉进来？

  • SEO / 内容策略
  • 付费广告动作
  • 社交媒体 & KOL
  • 渠道 & 分发入口
  • PLG / 免费策略
  • 定价杠杆变化

🟡 P1 · 留存情报

竞品怎么让人留下来？

  • 版本 & 功能发布
  • 产品能力方向
  • 用户痛点修复
  • 技术护城河变化
记录边界：只写来源能够直接证明的产品变化
禁止写入：无法由来源直接证明的目的、目标人群、战略意图、预期效果；周期外事件不进入正文时间线。

GIQ — Growth Intelligence Questions

GIQ 只定义调查覆盖面，不允许替代证据或引导主观推断。

# 示例 GIQ（每个竞品 3-5 个） GIQ-01: 竞品本周期在 SEO 侧发布或更新了哪些内容/落地页？ GIQ-02: 竞品有没有新的付费广告素材或落地页变化？ GIQ-03: 竞品有没有开始做 Influencer/KOL 合作？ GIQ-04: 竞品的免费策略/定价有没有变化？ GIQ-05: 竞品本周期公开发布了哪些产品能力变化？

04 · 双信源核验

确认状态与证据边界

✅ 已二次确认 两份相互独立的证据共同支持事件核心事实
⚠️ 部分确认 二次信源仅确认部分内容，或只能确认页面更新时间而非首次上线时间
◯ 单一来源 / ❌ 来源冲突 只有一个有效来源时保留事件但降级；来源冲突时并列记录差异，不自行裁决
二次信源独立性：
只有不同组织或独立发布者形成的第二份原始证据，才能标记“已二次确认”。官方文章 + 官方产品页、官方社媒 + 官方 sitemap 均属于同源佐证，最高标记“部分确认”；同一新闻稿转载、缓存副本和聚合复述不算二次确认。

09 · 实现方式

基于 Skill 的 Agent 驱动架构

核心原则：不写固定爬虫。采集层由 Agent 根据平台选择可用的浏览、搜索和结构化接口；SKILL.md 规定必查信源、最低动作、核验边界和台账门槛。

参考模式：hv-analysis skill

设计维度hv-analysiscompetitor_intel
Skill 形式纯 SKILL.md 指令文件，零代码同，纯 SKILL.md
采集方式搜索发现 → 页面提取 → 浏览器核验动态页按平台选择可用工具并保留执行台账
并行策略子 Agent 按信息类型并行子 Agent 按信源并行
触发方式一次性深度研究每两周定期运行
状态管理无（每次全量）state 仅作采集缓存，报告按固定周期独立调查
采集目标开放式信息收集GIQ 驱动定向采集
输出PDF 研究报告Markdown 双周报告落盘

工具链原则

首选
真实浏览器

用于动态渲染、登录态、广告透明度和社媒时间线；访问失败要记录具体原因。

次选
静态页面或结构化接口

用于官网、sitemap、RSS、Reddit、HN、公开 Release 和评论平台可访问页面。

补充
搜索引擎补查

使用品牌、别名、域名、产品词和动作词组合发现导航外页面与第三方原始内容。

完整项目文件结构

# Skill 指令层（user-level，跨项目可用） ~/.claude/skills/competitor_intel/ └── SKILL.md # agent 读取并自主执行的唯一入口 # 数据层（项目内，归版本管理） tools/competitor_intel/ ├── design.html # 本设计文档 ├── config/ │ └── competitors.yaml # 竞品 + GIQ + 信源配置 ├── state/ │ └── {competitor}.json # 可选采集缓存，不作为新增证据 └── reports/ └── {competitor}_{date}.md # 双周报告，由 agent 落盘

SKILL.md 核心指令结构

Step 1 确定双周窗口 → 读 competitors.yaml，获取目标竞品、GIQ、信源列表 → 以报告结束日向前 14 天建立独立调查区间 Step 2 子 Agent 并行采集（按信源拆分） → A：官网、Blog、产品、Pricing、Help、Changelog、Sitemap、RSS → B：X、YouTube、LinkedIn、TikTok、Instagram → C：Reddit、HN、评论平台和第三方 Creator → D：Product Hunt、Marketplace、合作方和媒体原创内容 → E：Meta / Google 广告透明度 → F：公开仓库、API/SDK、扩展商店和 Status Page（如适用） Step 3 执行台账 + 事件账本 + 二次核验 → 每个必查渠道记录入口、检索动作、候选数、状态和失败原因 → 合并候选事件，记录事实、发生时间和时间依据 → 判断独立信源/同源佐证/转述副本，再标记确认状态 Step 4 时间排序 + 输出报告 → 时间线在前，结构化事件卡居中 → 底部输出信源总表、采集执行台账、GIQ 覆盖和缺口
